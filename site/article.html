Why AI agents forget, contradict each other, and why nothing is auditable — and what RECALL does about it.

THE MISSING PIECE

June 2026

Building one AI agent is trivially easy now. Getting a group of them to actually remember anything, agree on anything, or be auditable afterward — that's the unsolved part.

The moment you have several agents collaborating on a real problem, three things break immediately.

They Forget

Every session starts blank. An agent that spent six months learning your customers, your edge cases, your patterns — gone the moment the context window closes. The next session, it starts from zero. You're not running an agent. You're running a very expensive stateless function.

They Contradict Each Other

Support approves a credit. Fraud flags the same customer two minutes later. Billing processes the credit anyway. None of them knew what the others did. Nobody wrote it down. The customer calls back tomorrow with the same problem, and the cycle starts again. This isn't a hypothetical — it's the default behavior of every multi-agent system in production today.

Nothing Is Auditable

When an agent makes a wrong decision, the answer to "why did it do that?" is almost always: we don't know. The memory that informed the decision lived in a context window, got flushed, and is gone. In regulated industries this is becoming a compliance crisis. Regulators are starting to ask for decision trails. Current agent architectures cannot provide them.

So I built the missing piece. It's called RECALL — a shared memory operating system for AI agents.

What RECALL Actually Is

RECALL isn't another agent framework. It sits underneath the agents you already have — any model, any framework, any team — and gives them one thing they don't have: a shared, governed, permanent memory layer.

Every agent reads the workspace before responding. Every agent writes to it after acting. The workspace becomes the single source of truth for everything that has happened with a given customer, case, or context. When agent B acts, it knows what agent A already did. When agent C contradicts agent B, RECALL catches it immediately — before damage, not after.

Four things make this work:

Shared workspaces — multiple agents, one memory context. Support, Fraud, and Billing all read from the same workspace. No more silos. No more agents working blind.

Conflict detection — when two agents write contradictory signals about the same entity, RECALL detects the contradiction, stores both writes permanently, and surfaces it for resolution. The higher-trust agent's signal takes precedence for auto-resolution, but nothing is silently overwritten. Every conflict is logged. Every resolution is receipted.

Cryptographic receipts — every memory write produces a content-addressed, signed, append-only receipt. Receipt ID is the SHA-256 of the canonical serialization. Two implementations observing the same action produce bit-identical receipt IDs. When someone asks "why did the agent make that decision" — you walk the receipt DAG. The answer is always there.

The registry — there's no npm for agent intelligence. A team spends six months training an agent. It learns edge cases, builds patterns, gets good. That work lives in a private codebase, undocumented, non-transferable. Another team solving the same problem starts from scratch. RECALL's registry fixes this. Publish your agent's accumulated memory once. Any team imports it and starts with full context — permanently stored on Walrus, cryptographically immutable. The quality guarantee is on-chain.

Why the Storage Choice Matters

Every design decision in RECALL flows from one principle: memory must be verifiable.

When an agent makes a decision, the memory that informed that decision must exist as a permanent, tamper-proof record that anyone authorized can inspect. This is impossible with centralized storage. A database entry can be modified. A log can be altered. A blob on Walrus cannot.

Walrus provides permanence — blobs are stored for their encoded duration, retrievable forever. Content-addressed storage means the blob ID is a cryptographic proof of content. Every memory write in RECALL is a Walrus blob. Every receipt batch has its Merkle root anchored on Sui. No agent data goes on-chain — only hashes. But those hashes mean any party can independently verify the full decision trail without trusting the control plane.

Governance rules live in Move contracts on Sui. The control plane cannot override them — they're verified on-chain. Rules like "PII cannot be written to external scope" or "low-trust agents require supervisor countersign" are not config files. They're enforced by the Move verifier.

What You'd Build With It

Customer operations — support, billing, and fraud agents sharing a workspace. Support resolutions inform billing decisions. Fraud flags prevent credits from being exploited. Compliance exports audit reports in minutes, not weeks.

DeFi coordination — research, risk, and execution agents where every order is traceable to the exact memory that informed it. Every market observation, every risk assessment, permanently receipted.

Healthcare triage — diagnostic agents where contradictory assessments are flagged for human review before any recommendation is made. The full decision trail available for medical audit.

Any regulated work — finance, insurance, legal — where you need to prove afterward exactly what happened, which agent decided it, what memory it was operating on, and under which governance rules.

The CLI Is the Product

Developers don't need a dashboard. They need answers. RECALL ships as a CLI first.

recall logs — stream every agent write in real time across all workspaces.

recall failures — surface every conflict and denied write instantly.

recall why --entity <id> — walk the full receipt trail for any entity.

recall inspect <id> — deep dive on any receipt, memory entry, or conflict.

recall export --entity <id> — generate a PDF audit trail for compliance.

The dashboard is just proof it works. The CLI is how you actually use it.
